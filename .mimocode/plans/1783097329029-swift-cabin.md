# Plan: Sécurisation CKBPOS / CKBPOS-PRO

## Objectif

Corriger les failles critiques de sécurité via des modifications de code ciblées. Phases 5 (git cleanup) et 6 (MDP admin aléatoire) mises de côté sur instruction utilisateur.

## Phase 1 — Blocage DevTools en production (CKBPOS + CKBPOS-PRO)

**Fichiers :**
- `CKBPOS\main.js` (lignes 104-111)
- `CKBPOS-PRO\main.js` (lignes 86-93)

**Changement :** Wrapping du handler `before-input-event` dans `if (isDev)` — F12 et Ctrl+Shift+I ne fonctionneront qu'en mode développement.

**Avant (identique dans les deux fichiers) :**
```js
mainWindow.webContents.on('before-input-event', (_, input) => {
  if (input.type === 'keyDown' && (
    input.key === 'F12' ||
    (input.control && input.shift && input.key === 'I')
  )) {
    mainWindow.webContents.toggleDevTools();
  }
});
```

**Après :**
```js
if (isDev) {
  mainWindow.webContents.on('before-input-event', (_, input) => {
    if (input.type === 'keyDown' && (
      input.key === 'F12' ||
      (input.control && input.shift && input.key === 'I')
    )) {
      mainWindow.webContents.toggleDevTools();
    }
  });
}
```

**Note :** `isDev` est déjà défini au-dessus dans les deux fichiers.

## Phase 2 — Garde `dev-sql-query` en production (CKBPOS-PRO uniquement)

**Fichier :** `CKBPOS-PRO\main.js` (lignes 472-485)

**Problème :** Aucune garde `isDev`. Exécute n'importe quel SQL (DROP, DELETE, INSERT...) en production via le renderer.

**Avant :**
```js
ipcMain.handle('dev-sql-query', (_, sql) => {
  try {
    const s = sql.trim();
    if (!s) return { success:false, error:'Requête vide' };
    const up = s.toUpperCase();
    if (up.startsWith('SELECT') || up.startsWith('PRAGMA')) {
      const rows = db.prepare(s).all();
      return { success:true, rows, count: rows.length };
    } else {
      const r = db.prepare(s).run();
      return { success:true, rows:[], count: r.changes, info: 'changes: ' + r.changes };
    }
  } catch(e) { return { success:false, error:e.message }; }
});
```

**Après :**
```js
ipcMain.handle('dev-sql-query', (_, sql) => {
  if (process.env.NODE_ENV === 'production' || !process.env.ELECTRON_IS_DEV) {
    return { success:false, error:'Non disponible en production' };
  }
  try {
    const s = sql.trim();
    if (!s) return { success:false, error:'Requête vide' };
    const up = s.toUpperCase();
    const BLOCKED = /^\s*(DROP|DELETE|TRUNCATE|ALTER|INSERT|UPDATE|CREATE)\b/;
    if (BLOCKED.test(up)) return { success:false, error:'Commande non autorisée en mode debug' };
    if (up.startsWith('SELECT') || up.startsWith('PRAGMA')) {
      const rows = db.prepare(s).all();
      return { success:true, rows, count: rows.length };
    } else {
      return { success:false, error:'Seules les requêtes SELECT/PRAGMA sont autorisées' };
    }
  } catch(e) { return { success:false, error:e.message }; }
});
```

## Phase 3 — Whitelist de tables pour `db-query` (CKBPOS-PRO uniquement)

**Fichier :** `CKBPOS-PRO\main.js` (lignes 340-451)

**Problème :** Aucune whitelist de tables. Le renderer peut exécuter `SELECT * FROM users`, `UPDATE users SET role='admin'`, etc.

**Étapes :**
1. Ajouter `DB_QUERY_WHITELIST` (copié de CKBPOS) avant le handler `db-query`
2. Ajouter le regex d'extraction de noms de tables
3. Bloquer les opérations DDL (DROP, ALTER, CREATE, TRUNCATE) au niveau du type de requête
4. Le reste du handler reste identique

**Bloc à insérer avant `ipcMain.handle('db-query', ...)`:**
```js
const DB_QUERY_WHITELIST = new Set([
  'users','products','product_variants','stock_mouvements','ventes','vente_items',
  'shifts','settings','clients','reservations','reservation_items','empresas',
  'caderno_entries','caderno_motivos','caderno_trabalhadores','caderno_produtos',
  'network_peers','sync_log','sync_state','stock_reservations','print_queue',
  'coordinator_log','user_sessions','document_series','fiscal_config',
  'lancamentos_contabilisticos','pgc_contas','historique_modifications',
  'notas_credito_debito'
]);
```

**Bloc à ajouter en début du handler :**
```js
// Whitelist tables
const tablePattern = /\b(?:FROM|INTO|UPDATE|JOIN)\s+["`]?(\w+)["`]?/gi;
let m;
while ((m = tablePattern.exec(sql)) !== null) {
  const tbl = m[1].toLowerCase();
  if (!DB_QUERY_WHITELIST.has(tbl)) {
    return { success:false, error:'Table non autorisée: ' + tbl };
  }
}
// Blocage DDL
const DDL_BLOCK = /^\s*(DROP|ALTER|TRUNCATE|CREATE)\b/;
if (DDL_BLOCK.test(sql.trim().toUpperCase())) {
  return { success:false, error:'Opération DDL non autorisée' };
}
```

## Vérification post-implémentation

1. **Syntaxe** : chaque fichier modifié doit être un JS valide (pas d'erreur de bracket, virgule, etc.)
2. **Phase 1** : `isDev` doit être défini AVANT le bloc `if (isDev)` — vérifier l'ordre dans les deux main.js
3. **Phase 3** : grep toutes les appels `dbQuery(...)` dans `src/` pour lister les tables utilisées et les comparer à la whitelist
4. **Phase 4** : les 6 handlers (3 par projet) doivent recevoir `user_id` comme integer, pas string

**Tests manuels :**
- F12 en production → rien ne se passe
- `electron.devSqlQuery('DROP TABLE users')` en production → erreur
- `electron.dbQuery('DELETE FROM users')` → bloqué (DDL) ou whitelisted (DELETE est autorisé mais limité aux tables de la whitelist)
- Vendeur appelant `cadernoEntriesList({is_admin:true})` → retourne uniquement ses propres entrées
