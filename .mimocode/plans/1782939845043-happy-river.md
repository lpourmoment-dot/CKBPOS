### Current Printing Method
**`webContents.print()` with `silent: true`** — HTML rendered in hidden BrowserWindow, sent to printer via Chromium print engine.

**No ESC/POS library. No thermal printer library. No `node-printer`. No raw data support.**

### Root Cause: Why Mobile Thermal Printers Produce an Empty Strip

1. **Custom page size incompatibility**: `pageSize: { width: 72100, height: 400000 }` (microns) is a non-standard page size. Desktop POS-80 printers with full Windows drivers handle this. Mobile thermal printers (Bluetooth/USB) via generic/WinUSB drivers do NOT — they either ignore the size or feed a minimal blank strip.

2. **No ESC/POS raw data**: Mobile thermal printers expect ESC/POS commands for paper feed, cut, and content. The HTML→Chromium→GDI→driver pipeline loses critical information when the driver doesn't understand the custom page dimensions.

3. **Silent print with no fallback**: When `webContents.print()` fails silently (success=false), the only fallback is to retry without `deviceName` — still the same broken approach.

4. **1500ms delay is printer-agnostic**: The fixed `setTimeout(…, 1500)` before printing doesn't adapt to printer warm-up time.

5. **No printer capability detection**: The system only gets printer name/default/status via `getPrintersAsync()`. No paper width, DPI, or ESC/POS capability detection.

## 2. Implementation Plan

### Phase 1: Create ESC/POS Engine (`src/utils/escpos.js`)

**New file** — Pure JS ESC/POS command generator (no npm dependency).

Commands to implement:
- `initialize()` → ESC @ (0x1B 0x40)
- `setFont(size)` → ESC ! (bold, double-width, double-height)
- `setAlignment(align)` → ESC a (0=left, 1=center, 2=right)
- `feedPaper(lines)` → ESC d (n lines)
- `cutPaper()` → GS V (full/partial cut)
- `printText(text)` → raw UTF-8
- `printLine(text)` → text + LF
- `setLineSpacing(dots)` → ESC 3
- `setCharacterSpacing(dots)` → ESC SP
- `setCodePage(codepage)` → ESC t (for accent support)
- `printBarcode(data, type)` → GS k
- `printQRCode(data)` → GS ( k
- ` rasterImage(bitmap)` → GS v 0 (for logo/QR as image)

### Phase 2: Create Printer Detection Module (`src/utils/printer-detect.js`)

**New file** — Windows printer capability detection.

Functions:
- `detectPrinterCapabilities(printerName)` → queries Windows via `wmic` or PowerShell for:
  - Printer driver name
  - Port (USB, COM, Bluetooth, network)
  - Paper widths supported
  - DPI
  - Whether it's a POS/thermal printer
- `classifyPrinter(printerInfo)` → returns `'escpos' | 'windows_driver' | 'pdf' | 'unknown'`
- `detectPrinterType(name)` → heuristic from printer name patterns (POS-80, Mobile, Bluetooth, MTP, etc.)

### Phase 3: Create Adaptive Print Engine (`src/utils/adaptive-print.js`)

**New file** — Multi-strategy print orchestrator.

Strategy selection:
```
1. If printer classified as ESC/POS capable → generate ESC/POS raw data → send via raw socket/USB
2. If printer is standard Windows → use current webContents.print() (existing behavior)
3. Fallback: generate PDF → print via Windows default printer
```

The engine will:
- Auto-detect printer type on first print
- Cache detection results
- Try strategy 1, if fail try strategy 2, if fail try strategy 3
- Log each attempt with timing

### Phase 4: Modify `printHTML()` in `main.js`

**File**: `main.js` (lines 1043-1154)

Changes:
1. **Add printer detection before printing** — call `detectPrinterCapabilities()` on first use
2. **Route to appropriate strategy** based on detection
3. **For ESC/POS printers**: bypass `webContents.print()` entirely, generate ESC/POS commands and send via raw data
4. **For Windows printers**: keep existing `webContents.print()` but with improved page size handling
5. **Add proper paper feed + cut commands** after content
6. **Better error handling and retry logic**
7. **Detailed logging** for diagnostics

### Phase 5: Add ESC/POS Ticket Generator (`main.js`)

**File**: `main.js`

New function `generateTicketESCPOS(data)` that converts the same ticket data to ESC/POS commands instead of HTML. This handles:
- Shop name (bold, centered, large)
- Separator lines
- Items table (aligned columns)
- Total (bold, large)
- Payment info
- QR Code (as raster image)
- Paper cut

### Phase 6: Enhanced Settings UI

**File**: `src/pages/SettingsPage.js`

Add:
- Printer type detection indicator (shows detected type)
- Manual override: Auto / ESC/POS / Windows Driver / PDF
- Paper width: Auto / 58mm / 80mm
- Connection type display (USB, Bluetooth, Network, Serial)
- "Test Print" button that prints a test page
- Detailed printer info display (driver, port, DPI, capabilities)

### Phase 7: Enhanced Preload IPC

**File**: `preload.js`

Add new IPC bridges:
- `detectPrinter()` — triggers capability detection
- `testPrint()` — prints a test page
- `getPrinterInfo()` — returns detailed printer info

### Phase 8: Enhanced Main IPC

**File**: `main.js`

Add new IPC handlers:
- `detect-printer` — detect printer capabilities
- `test-print` — print test page
- `get-printer-info` — return detailed printer info
- Update `get-printers` to include more details

### Phase 9: Settings DB Schema

Add new settings keys:
- `printer_method` → 'auto' | 'escpos' | 'windows' | 'pdf'
- `printer_paper_width` → 'auto' | '58' | '80'
- `printer_connection` → 'auto' | 'usb' | 'bluetooth' | 'network' | 'serial'
- `printer_capabilities` → JSON cache of detection results

### Phase 10: Enhanced Logging

Add structured logging for print operations:
- Printer detected: name, type, driver, port, capabilities
- Strategy chosen and why
- Data size sent
- Print time (start → finish)
- Success/failure with error details
- Fallback attempts
