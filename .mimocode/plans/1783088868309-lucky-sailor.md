# Plan : Refactorisation de main.js + Unification des Traductions

## Contexte

CKBPOS a deux problèmes architecturaux majeurs qui freinent la maintenance et introduisent des bugs silencieux :

1. **main.js** fait 5 182 lignes dans un seul fichier avec 114 IPC handlers, du monkey-patching, 690 lignes de templates HTML, et de la logique métier dupliquée
2. **Le système de traductions** a 3 sources (translations.js, CADERNO_T dans CadernoPage, chaînes hardcodées) avec ~150+ clés mortes et des doublons silencieux

---

### Risques et mitigation

| Risque | Mitigation |
|--------|------------|
| Casser les IPC channels existants | Chaque module extrait enregistre exactement les mêmes channel names |
| Cycler les imports (db → module → db) | Passer `db` en paramètre de `register(ipcMain, db, store)` |
| Le monkey-patching de sync se casse | Étape 2.4 le remplace par une chaîne explicite |
| Les templates HTML cassent | Fonctions pures, testées indépendamment |
| Les timers leak | Chaque module gère ses propres timers et les clear proprement |

### Étapes d'implémentation

#### Étape 1 : Nettoyer translations.js (~2 950 lignes actuelles)

**1.1 — Supprimer les ~150+ clés en double**

Dans les 3 langues, les sections `settings`, `login`, `cashier`, `users` contiennent des clés définies 2+ fois. La dernière définition gagne silencieusement. Il faut :
- Identifier toutes les clés en double (le parser AST ou un script peut le faire)
- Garder la version la plus récente (celle qui est effectivement lue)
- Supprimer les versions antérieures

Sections concernées (approximativement) :
```
settings : ~30+ doublons par langue
login    : ~2 doublons par langue  
cashier  : ~3 doublons par langue
users    : ~1 doublon par langue
→ Total : ~36 x 3 langues = ~108 clés mortes à supprimer
```

**1.2 — Corriger le bug d'imbrication `caderno` dans `settings`**

Dans les 3 langues, `caderno` est défini au top-level ET imbriqué dans `settings` :

```javascript
// AVANT (bug) :
export const translations = {
  'pt-BR': {
    caderno: { /* clés caderno */ },        // ← top-level (marche)
    settings: {
      // ...
      caderno: { configTitle: '...' },       // ← imbriqué (bug)
    },
  }
};

// APRÈS (corrigé) :
export const translations = {
  'pt-BR': {
    caderno: { /* toutes les clés caderno unifiées ici */ },
    settings: { /* sans la section caderno imbriquée */ },
  }
};
```

## Ordre d'implémentation recommandé

1. **Traductions** d'abord (plus simple, impact immédiat, zéro risque de casser les IPC)
   - Étape 1.1-1.3 : nettoyer translations.js
   - Étape 2 : modifier CadernoPage.js
   - Étape 3 : nettoyer useLang.js
   - Étape 4 : vérifier

2. **main.js templates** ensuite (fonctions pures, extraction facile)
   - Étape 1.1-1.3 du partie 1 : templates

3. **Modules indépendants** puis (store, console, updater, backup, drive, empresas)
   - Étape 1.4-1.6 + 3.5-3.7 + 3.13 du partie 1

4. **Modules réseau** ensuite (les plus complexes, monkey-patching)
   - Étape 2.1-2.4 du partie 1

5. **Modules IPC métier** enfin (dépendent de db et réseau)
   - Étape 3.1-3.4 + 3.8-3.12 + 3.14 du partie 1

6. **Réécriture main.js** en orchestrateur
   - Étape 4 du partie 1

## Vérification

Après chaque étape :
1. `npm run dev` — l'app doit démarrer et fonctionner normalement
2. `node scripts/verify-build.js` — doit passer toutes les phases
3. Tester manuellement : caisse, produits, historique, caderno, settings, messagerie
4. Après la partie 1 complète : `npm run dist` — le build doit réussir
5. Vérifier que `app.asar` contient tous les modules extraits
6. Vérifier que les traductions fonctionnent dans les 3 langues
